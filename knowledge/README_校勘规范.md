# 玄照知识库 — 校勘规范

> 本规范为**玄照 v2 知识库**所有 markdown 文件的统一标准。
> 适用对象: 任何 markdown 文件 `knowledge/data/**/*.md`
> 起草日: 2026-07-09
> 起草者: Claude (梧助手)

---

## 一、目标

让玄照知识库**从"速记表"升级为"可被专家信任的知识库"**。

定义可被专家信任:
- **可追溯**: 每个论断有**古籍/论文/案例**出处
- **可校验**: 文件中提到的格局/星曜/神煞,能在玄照引擎(已实现代码)中**重放**
- **可分层**: 内容按 **核心常识 / 流派差异 / 学术争议** 分级
- **可更新**: 单条修改有 commit 记录,**不批量覆盖**

---

## 二、内容分层

每个 markdown 文件**必须**分三段:

### L1 - 核心要点 (速记)

- 一行一概念
- 适合快速检索
- 例:`天乙贵人: 甲戊庚→丑未,乙己→子申,丙丁→亥酉...`

### L2 - 完整体系 (知识)

- 表格化
- 列出**完整**体系(如 14 星 × 12 宫,45 神煞全表)
- 出处标注在前/末
- 例:14 主星庙陷利陷表

### L3 - 流派与争议 (深度)

- 不同流派的差异
- 现代研究的不同看法
- **明确立场**(玄照采纳哪种,为什么不采纳其他)

---

## 三、出处标注规范

每个 markdown 文件**必须在 frontmatter 或文末**包含 `sources` 字段:

```markdown

last_updated: 2026-07-09
audit_status: "已校勘" | "待校勘" | "已弃用"

---
```

## 六、提交规范

每次 commit 必须**遵守**:

### commit message 格式

```
feat(术法): 简述

- 新增 X (源: 《Y》卷 Z)
- 补全 A → B (源: 《C》)
- 修正 D (源: 《E》)

审计状态: 已校勘 → 下一轮校勘点
```

### 反例(禁止)

- ❌ `feat: enhance engine`
- ❌ `补充知识库`
- ❌ `evolution #1: 2026-...`

---

## 七、与 auto-evolution 的边界

auto-evolution 类系统**禁止**直接修改 `knowledge/data/**/*.md`:
- 修改必须由**有真实古籍/资料引用的 agent 或人**做出
- 修改必须**有 commit message 中的 sources**
- 修改后必须**经过至少一次人工/agent 抽检**
- 违反将视为破坏数据可靠性
